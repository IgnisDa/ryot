# God Mode Auth Recovery

## Problem Statement

Ryot V1 allowed users to authenticate with either a local password or OIDC. V1 also had an admin-user concept that could create password reset links for other users. V2 uses Better Auth, stores local credentials in Better Auth accounts, and intentionally does not port V1 admin users. The current legacy bootstrap migrates V1 users into V2 user rows but does not create credential accounts, so migrated password users cannot sign in. V1 OIDC users also need their identity link preserved so that they continue signing in through the same OIDC issuer without creating duplicate empty users.

The product needs a clean migration and recovery path that avoids storing legacy password hash formats in V2, preserves V1's local-vs-OIDC mutual exclusivity, and replaces V1 admin users with a server-token-protected administrative surface.

## Solution

Add a `god-mode` administrative surface owned by the V2 backend and exposed through the app client. The backend accepts `SERVER_ADMIN_ACCESS_TOKEN` via a bearer token and provides admin-only user listing and password reset link generation. The app client exposes a UI route for self-hosted admins to paste the admin token, view auth status for users, and generate reset links for password-backed users. Cloud customer-owned reset link generation reuses the same backend operation from the website account page.

The legacy bootstrap should migrate user rows, reject corrupt auth states, avoid migrating old password hashes, and create minimal Better Auth OIDC account stubs for legacy OIDC users. Password-backed migrated users intentionally start with no credential account and must receive a reset link through `god-mode`. OIDC-backed migrated users keep OIDC-only access and cannot receive local password reset links.

Better Auth remains the owner of all normal auth operations. Outside the legacy bootstrap migration, code must not manually mutate Better Auth-managed tables. Password reset links are generated by Better Auth's password reset API. The backend captures the reset URL through Better Auth's `sendResetPassword` callback and Redis, then returns it to the caller.

## User Stories

1. As a self-hosted admin, I want to open a god-mode route, so that I can recover migrated family member accounts without a V1 admin user.
2. As a self-hosted admin, I want to paste `SERVER_ADMIN_ACCESS_TOKEN` into god-mode, so that administrative operations are protected by an instance-owned secret.
3. As a self-hosted admin, I want god-mode to avoid storing my admin token, so that a browser reload or local storage leak does not retain the secret.
4. As a self-hosted admin, I want to see a list of users, so that I can find the account that needs recovery.
5. As a self-hosted admin, I want user search/filtering by email or name, so that I can find accounts in larger instances.
6. As a self-hosted admin, I want each listed user to show whether they authenticate with local password, OIDC, no credential yet, or an invalid mixed state, so that I know which recovery actions are valid.
7. As a self-hosted admin, I want to generate a password reset link for a migrated password user with no credential account, so that the user can create a V2 password.
8. As a self-hosted admin, I want to generate a password reset link for an existing V2 password user, so that the same tool remains useful after migration.
9. As a self-hosted admin, I want OIDC users to be blocked from local password reset links, so that V1's mutually exclusive auth model remains intact.
10. As a self-hosted admin, I want users with mixed OIDC and credential accounts to be flagged as invalid, so that the system does not silently normalize corrupt auth state.
11. As a self-hosted admin, I want reset links to expire with Better Auth's default reset expiry, so that links remain short-lived and user-driven.
12. As a self-hosted admin, I want generated reset links to be copyable/shareable, so that I can send them to family members manually.
13. As a self-hosted admin, I want reset link generation to fail when local auth is disabled, so that I do not hand out links that cannot be used.
14. As a migrated password user, I want the reset link to take me directly to a reset password page, so that I can set a V2 password without knowing V1 internals.
15. As a migrated password user, I want my new password to be stored using Better Auth's normal password hashing, so that V2 does not carry legacy Argon2 hashes.
16. As a migrated password user, I want to sign in with the same email shown by the reset tooling, so that I know which identifier to use after setting my password.
17. As a migrated OIDC user, I want my existing OIDC sign-in to keep working after migration, so that I do not need a password reset.
18. As a migrated OIDC user, I want local password reset to be unavailable for my account, so that OIDC remains my only auth provider as in V1.
19. As a cloud customer, I want the website account page to generate a reset link only for my own app user, so that customer-scoped recovery does not require god-mode UI access.
20. As a cloud customer using OIDC, I do not want a password reset link, so that my cloud account preserves OIDC-only access.
21. As an operator, I want legacy migration to fail loudly on users with both password and OIDC identity, so that corrupt auth state is not silently migrated.
22. As an operator, I want legacy migration to fail loudly on users with neither password nor OIDC identity, so that authless accounts are not created accidentally.
23. As an operator, I want legacy migration to fail loudly when an OIDC user does not have a valid email-style name, so that unexpected V1 identity data is caught before old tables are dropped.
24. As an operator, I want OIDC account stubs to be restart-safe, so that rerunning legacy bootstrap does not duplicate links.
25. As an operator, I want OIDC account stubs to contain no stale tokens, so that V2 only stores fresh tokens after the user's next OIDC sign-in.
26. As an operator, I want missing OIDC runtime config to remain a runtime concern, so that migration can preserve identity links without enforcing deployment configuration at migration time.
27. As a developer, I want god-mode to use Better Auth's server SDK/context for listing and reset operations, so that auth behavior remains aligned with Better Auth.
28. As a developer, I want no direct writes to Better Auth tables outside legacy bootstrap, so that normal auth state remains owned by Better Auth.
29. As a developer, I want OpenAPI contracts for god-mode endpoints, so that the app client can call them through generated API types.
30. As a developer, I want backend unit tests for auth state classification and reset eligibility, so that edge cases are covered without brittle UI tests.
31. As a developer, I want E2E coverage for god-mode reset flow, so that a user can generate a reset link, set a password, and sign in.

## Implementation Decisions

### Modules to build or modify

- Build a backend `god-mode` module with schemas, service logic, route definitions, and unit tests.
- Modify the backend auth instance to enable Better Auth password reset callbacks, revoke sessions on password reset, and publish internally generated reset links to Redis for god-mode capture.
- Modify legacy bootstrap user migration to validate V1 auth states, preserve password users without credential accounts, and create minimal OIDC account stubs for OIDC users.
- Modify legacy bootstrap documentation to reflect the final decisions: password credentials are reset-only, OIDC identity links are migrated, 2FA is not migrated, V1 admin users are not ported.
- Add app-client public routes for god-mode and reset-password.
- Add a website account action for cloud customer-scoped password reset link generation using the backend god-mode operation.
- Add backend unit tests for god-mode service behavior.
- Add E2E tests for admin token enforcement, reset link generation, reset completion, and OIDC blocking.

### Legacy bootstrap decisions

- Legacy user IDs remain preserved in V2 user rows.
- Valid email-style legacy names remain V2 user emails. Non-email names become deterministic `@ryot.local` fallback emails with collision handling.
- The email validation regex must require a literal dot before the top-level domain.
- Legacy users with both a non-empty password and a non-empty OIDC subject are invalid and must abort migration.
- Legacy users with neither a non-empty password nor a non-empty OIDC subject are invalid and must abort migration.
- Legacy OIDC users whose legacy name is not a valid email must abort migration.
- Legacy password hashes must not be inserted into Better Auth credential accounts.
- Legacy password users should migrate as V2 users without credential accounts. They are recovered by god-mode reset links.
- Legacy OIDC users should receive one minimal Better Auth account stub with provider ID `oidc`, account ID equal to the legacy OIDC subject, user ID equal to the migrated user ID, and null token/password fields.
- OIDC account stub IDs must be deterministic so `ON CONFLICT DO NOTHING` keeps the migration restart-safe.
- Legacy OIDC account stubs should not include access tokens, refresh tokens, ID tokens, token expiry, scope, image, or profile metadata. Better Auth refreshes account token fields on the next successful OIDC sign-in.
- Migration should not fail solely because OIDC runtime configuration is missing. Preserving identity links is migration work; actual sign-in depends on deployment configuration.
- Fail-fast checks should verify that every legacy OIDC user has a matching migrated account stub before old tables are dropped.
- Fail-fast checks should verify that no credential accounts were created for migrated password users during legacy bootstrap.

### Backend god-mode API contract

- All god-mode endpoints require `Authorization: Bearer <SERVER_ADMIN_ACCESS_TOKEN>`.
- Missing, malformed, or incorrect bearer tokens return an unauthenticated/unauthorized error without leaking whether the token was close.
- The user listing endpoint returns a paginated list of users and auth state summaries.
- User listing supports search by email or name and should keep the query shape small enough for app-client UI needs.
- User listing returns user ID, name, email, created timestamp, two-factor flag, and an auth state.
- Auth state values are `credential`, `oidc`, `none`, and `mixed`.
- `credential` means the user has a Better Auth credential account and no OIDC account.
- `oidc` means the user has an OIDC account and no credential account.
- `none` means the user has no Better Auth account rows. This is expected for migrated password users before reset.
- `mixed` means the user has both credential and OIDC accounts, or otherwise has a provider combination that god-mode should not normalize.
- User listing must not expose account IDs, OIDC subjects, access tokens, refresh tokens, ID tokens, passwords, scopes, or provider-specific secrets.
- The password reset link endpoint accepts a target user ID and returns a reset URL, expiry description if available, and the login email.
- Password reset link generation is allowed for `credential` and `none` users only.
- Password reset link generation is denied for `oidc` and `mixed` users.
- Password reset link generation is denied when local auth is disabled.
- Password reset link generation uses Better Auth's password reset API rather than direct verification table writes.
- Password reset completion uses Better Auth's existing reset-password operation. If the user has no credential account, Better Auth creates one. If the user already has a credential account, Better Auth updates it.

### Better Auth SDK/context usage

- Do not enable Better Auth's admin plugin for this feature. Ryot V2 intentionally does not model admin users in Better Auth roles.
- Use Better Auth's server context/adapter for user listing, total counts, and per-user account lookup. This satisfies the requirement to use Better Auth's server SDK while avoiding direct Drizzle reads for normal god-mode operations.
- Use Better Auth's password reset API for reset token creation and credential account creation/update.
- Do not directly mutate Better Auth-managed tables outside the legacy bootstrap migration.
- Read-only auth state classification should be implemented through Better Auth context/adapter, not hand-written SQL.

### Password reset link capture

- Configure Better Auth email/password reset support with a `sendResetPassword` callback.
- Configure password resets to revoke existing sessions after a successful password reset.
- God-mode reset generation creates a high-entropy request correlation ID.
- God-mode subscribes to a Redis channel or equivalent Redis-backed wait primitive before calling Better Auth's password reset API.
- God-mode calls Better Auth's password reset request with the target user's email and a redirect URL containing the request correlation ID.
- `sendResetPassword` receives Better Auth's generated token and URL, extracts the request correlation ID, builds the app-client reset URL, and publishes the generated reset URL back through Redis.
- The reset URL returned to the UI should point directly at the app-client reset-password route with the token in the query string.
- God-mode waits for the Redis response with a short timeout and returns a clear error if the reset callback is not observed.
- If Better Auth's public request-password-reset endpoint is called without a god-mode correlation ID, the callback should not expose or send a link.
- Redis capture must be safe across multiple backend processes, because cloud and self-hosted deployments may run more than one backend instance.
- The admin token must never be placed in query params, reset URLs, Redis payloads, logs, or local storage.

### App-client god-mode UI

- Add a public god-mode route.
- If the app client has no configured server URL, redirect to the existing auth flow rather than rebuilding server URL selection in god-mode.
- The admin token is entered into the god-mode page and held in React memory only.
- The page lists users through the god-mode backend API.
- The page clearly labels auth state for each user.
- The reset action is enabled for `credential` and `none` users and disabled for `oidc` and `mixed` users.
- The reset action displays the returned reset link and login email.
- The UI should provide a copy/share affordance for the reset link.
- The UI should surface backend validation errors directly enough for admins to understand why an action is blocked.
- God-mode UI should not expose OIDC subject/account IDs.

### App-client reset-password UI

- Add a public reset-password route.
- The route reads the reset token from the query string.
- If the token is missing, show a clear invalid-link state.
- The route accepts a new password and confirmation.
- The route calls Better Auth's reset-password client operation.
- On success, redirect the user to the auth route and tell them to sign in with the displayed/known email.
- On failure, show the Better Auth error message or a safe generic invalid/expired link message.

### Website account flow

- Add a customer-scoped action on the website account page for cloud users with an associated app user ID.
- The website action calls the backend god-mode reset endpoint server-side using `SERVER_ADMIN_ACCESS_TOKEN`.
- The website action is only allowed for the authenticated customer's own app user ID.
- The website action must not expose reset links for OIDC customers.
- The website action should reuse backend god-mode restrictions rather than duplicating auth-state rules.

## Testing Decisions

- Tests should verify externally observable behavior and stable service decisions, not Better Auth internals.
- Legacy bootstrap migration changes should follow the existing module guidance: validate by restoring legacy dumps, running the legacy bootstrap flow, and inspecting migrated rows. Do not add automated tests inside the legacy bootstrap module unless the module guidance changes.
- Backend unit tests should cover god-mode auth-state classification using stubbed Better Auth account data.
- Backend unit tests should cover reset eligibility for `credential`, `none`, `oidc`, and `mixed` states.
- Backend unit tests should cover admin bearer token parsing and rejection behavior.
- Backend unit tests should cover Redis reset link capture success, timeout, and missing correlation ID behavior using small fakes where possible.
- E2E tests should cover god-mode endpoint rejection without the admin bearer token.
- E2E tests should cover generating a reset link for a password/no-account user, completing the reset via Better Auth, and signing in with the new password.
- E2E tests should cover that OIDC-only users cannot receive local password reset links.
- E2E tests should cover that mixed auth users cannot receive local password reset links if a mixed state can be seeded safely.
- Prior art for backend unit tests includes service tests in existing backend modules that isolate business rules from routes.
- Prior art for E2E tests includes existing auth and OIDC tests that use HTTP calls, real backend processes, and database assertions.
- OpenAPI-generated types are regenerated by the dev backend server and should be consumed by app-client API calls once endpoints exist.

## Out of Scope

- Migrating V1 password hashes or adding an Argon2 verifier to V2.
- Porting V1 admin users or V1 admin roles into V2 Better Auth users.
- Enabling Better Auth's admin plugin for this feature.
- Creating local password login for OIDC-only users.
- Emailing reset links from the self-hosted god-mode UI.
- Public forgot-password UI outside god-mode and customer-scoped website flows.
- Changing Better Auth-managed table schemas or indexes.
- Migrating V1 sessions, V1 access tokens, V1 two-factor data, or V1 OIDC tokens.
- Updating user profile image/name from OIDC beyond Better Auth's default behavior.
- Supporting multiple OIDC providers.
- Rebuilding server URL configuration inside god-mode.

## Further Notes

- The same OIDC issuer/provider is expected to be used after migration. If an operator changes OIDC issuer, OIDC subjects may no longer match and those users will need operational recovery outside this PRD.
- Better Auth updates OAuth account token fields after successful OIDC sign-in. It does not update the V2 user's name/image unless override behavior is enabled, and this PRD intentionally avoids adding that behavior.
- Adding `sendResetPassword` enables Better Auth's reset request endpoint. Ryot's callback should only publish internally when a god-mode correlation ID is present; otherwise no reset link should be sent or exposed.
- Self-hosted migration documentation should explain that old local passwords intentionally stop working after V2 migration and admins must issue reset links from god-mode.
- The legacy bootstrap regex bug for valid email detection has already been corrected so the dot before the top-level domain is literal.

# New API Endpoints

**Parent Plan:** [Sandbox Async Redesign](./README.md)

**Type:** AFK

**Status:** todo

## What to build

Replace the single synchronous `POST /sandbox/run` endpoint with two new endpoints that implement the async enqueue-and-poll contract. After this slice the full sandbox API is functional end-to-end with the existing three host functions.

Specifically:

- Remove `POST /sandbox/run` and its route handler entirely.
- Add `POST /sandbox/enqueue`:
  - Request body: `{ code: string, context?: Record<string, unknown> }`. Keep the existing `code` max-length constraint (20,000 chars).
  - Route handler builds `apiFunctionDescriptors` server-side (for `httpCall`, `getAppConfigValue`, `getUserConfigValue` — all with empty context). The client never supplies descriptors.
  - Calls `sandboxService.enqueue(...)` and returns `{ data: { jobId: string } }`.
- Add `GET /sandbox/result/:jobId`:
  - Auth required.
  - Fetches the BullMQ job via `queue.getJob(jobId)`. Returns `404` if the job does not exist or if `job.data.userId` does not match the authenticated user.
  - Maps BullMQ job state (`waiting`, `active`, `delayed` → `"pending"`; `completed` → `"completed"`; `failed` → `"failed"`) to the response.
  - Completed response: `{ data: { status: "completed", logs, value, error } }` from `job.returnvalue`.
  - Failed response: `{ data: { status: "failed", error: string } }` from `job.failedReason`.
  - Pending response: `{ data: { status: "pending" } }`.
- Write unit tests for the poll endpoint covering all status branches and auth cases.

See the **API contract** section of the parent PRD for the full response shapes. See the **Error model** section for the script-error vs infra-error distinction.

## Acceptance criteria

- [ ] `POST /sandbox/run` no longer exists and returns `404` if called.
- [ ] `POST /sandbox/enqueue` accepts `{ code, context? }` and returns `{ data: { jobId } }` where `jobId` is the UUID generated by `enqueue()`.
- [ ] `POST /sandbox/enqueue` rejects unauthenticated requests with `401`.
- [ ] `GET /sandbox/result/:jobId` returns `{ data: { status: "pending" } }` for a queued-but-not-yet-complete job.
- [ ] `GET /sandbox/result/:jobId` returns `{ data: { status: "completed", logs, value, error } }` for a completed job.
- [ ] `GET /sandbox/result/:jobId` returns `{ data: { status: "failed", error } }` for a failed job, using BullMQ `failedReason`.
- [ ] `GET /sandbox/result/:jobId` returns `404` for a non-existent job ID.
- [ ] `GET /sandbox/result/:jobId` returns `404` (not `403`) when the job belongs to a different user.
- [ ] `GET /sandbox/result/:jobId` rejects unauthenticated requests with `401`.
- [ ] Unit tests cover all poll endpoint branches with a mock BullMQ job object.
- [ ] OpenAPI spec is regenerated to reflect the removed and added endpoints.
- [ ] `bun run typecheck`, `bun test`, and `bun run lint` pass in `apps/app-backend`.

## User stories addressed

- User story 12 (enqueue returns `jobId` immediately)
- User story 13 (stable `pending | completed | failed` poll response)
- User story 14 (completed response includes `logs`, `value`, `error`)
- User story 15 (failed response includes infrastructure error message)

- User story 17 (wrong-user job → `404`, not `403`)
- User story 18 (random UUID job IDs prevent enumeration)
- User story 22 (script-level error → `completed` with `error` set, not `failed`)
